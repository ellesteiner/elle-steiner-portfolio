Connection Indicator Component

Kessel Run - US Air Force Software

Role: Lead Designer  //  Timeline: 4 months  //  Team: 2 designers, 5 engineers  //  Impact: 7 applications

Led design and implementation of a standardized connection indicator component across 7 Air Force applications, reducing user confusion about connection status by 75% and establishing reusable patterns adopted throughout the design system.

Key Outcomes:

  • 75% reduction in connection-related support tickets
  • 7 applications successfully integrated the component in 3 months
  • User satisfaction improved 28% in post-deployment surveys

The Problem

User Impact

Warfighters using our mission-critical software suite experienced significant confusion around network connectivity status. With seven different applications each displaying connection information differently (or not at all), users couldn't confidently determine whether data was current or stale.

Pain Points Identified:

  • "I don't know if I'm seeing live data or if my connection dropped"
  • "Each app shows connection status differently, I can't tell what's actually happening"
  • "The app stops working and I don't know if it's my network or a bug"
  • "I make mission-critical decisions on this data, I need to know it's current"

Business Impact

Support ticket analysis revealed connection-related confusion was consuming significant resources:

  • 20+ monthly support tickets related to connection status across 7 apps
  • Average 45-minute resolution time per ticket
  • User trust issues impacting satisfaction scores
  • Inconsistent experiences across product suite

Technical Complexity

The challenge wasn't just design - it was technical:

  • Applications had different connectivity requirements (real-time vs. periodic sync)
  • Network conditions varied dramatically (tactical field operations vs. secure facilities)
  • Some apps required offline functionality with sync queues
  • Existing solutions in 5 apps were all implemented differently

My Approach

Phase 1: Research & Discovery

I assembled a cross-functional working group including representatives from all 7 product teams, backend engineers, subject matter experts (warfighters), and security team members. Through interviews with 12 warfighters, analysis of 50+ support tickets, and field observations during exercises, I uncovered the critical insight: users needed to understand what connection status meant for their specific workflow, not just binary connectivity.

Key Decision: Framed the problem around context-specific needs (warfighter needs current targeting data, analyst needs latest intel updates) rather than generic connection indicators. This insight drove all subsequent design decisions.

Research synthesis with the team identified 3 distinct connection states users needed to differentiate, 5 mission-critical scenarios where connection status impacted decisions, and 4 edge cases where existing solutions failed.

Phase 2: Collaborative Design

I partnered with the development team from day one, holding weekly design-dev sessions to gather technical feasibility feedback and understand backend connectivity constraints. This revealed that different apps had fundamentally different technical capabilities.

Key Decision: Designed a three-tier progressive enhancement system—Basic (binary connected/disconnected), Enhanced (connection quality), and Advanced (sync status with timestamp). This allowed rapid integration across all 7 apps while enabling teams to enhance based on specific needs rather than forcing a one-size-fits-all solution.

I tested 3 design directions with 8 warfighters. The winning solution (icon + color + text label) achieved 94% accuracy and WCAG compliance, compared to 62% for icon-only. Accessibility was non-negotiable: I ensured color wasn't the sole differentiator, added clear text labels for screen readers, and tested with keyboard navigation.

Phase 3: Development Partnership

I created comprehensive documentation covering component API specifications, visual design specs, interaction behaviors, edge case handling, and accessibility requirements. During the 3-month integration period, I held daily office hours to answer implementation questions from 7 product teams and reviewed pull requests for design consistency.

Key Decision: When one team's backend couldn't distinguish between "poor connection" and "timeout," I redesigned the indicator to combine these states with clear user messaging rather than forcing a technical solution—this actually improved the experience and prevented implementation delays.

The detailed specifications and real-time support enabled teams to integrate the component in an average of 2 days versus the previous 2-week timeline for similar features.

Phase 4: Launch & Iteration (Month 4)

Design Solution

Component Architecture

Three Connection States:

1. Connected (Green)

  • Visual: Green dot + "Connected" label
  • User understanding: "My data is current and I'm receiving live updates"
  • Technical state: Active websocket connection with recent heartbeat

2. Partially Connected (Yellow)

  • Visual: Yellow dot with rotating animation + "Partially Connected" label
  • User understanding: "Connection was lost, app is attempting to reconnect"
  • Technical state: Connection dropped, retry logic active, using cached data

3. Critical Connection (Red)

  • Visual: Red dot + "Critical Connection" label + last sync timestamp
  • User understanding: "No connection, viewing cached data from [time]"
  • Technical state: Connection failed, operating in offline mode

Interaction Design:

  • Hover: Displays detailed connection information tooltip
  • Click: Opens connection status modal with troubleshooting steps
  • Transitions: Smooth 300ms fade between states (not jarring)

Visual Design Decisions

Placement in application header (top-right) — User testing showed 95% discoverability versus 62% for footer placement. Users consistently checked the header for status indicators.

Color + text strategy — Used semantic colors (green/yellow/red) matching mental models, but added text labels to ensure accessibility. All color combinations met WCAG AA 4.5:1 contrast ratio minimum and were tested with colorblind simulation tools.

Size and prominence — Designed to be visible without distracting during normal use, with subtle pulse animation during state changes to draw attention when needed.

Collaboration & Leadership

Cross-Functional Coordination

I led collaboration across 7 product teams, mentoring a junior designer on component variants for Phase 2 enhancements while they successfully delivered new features. By establishing a shared communication channel, creating living documentation, and participating in engineering standups, I built trust through technical understanding and responsiveness.

Stakeholder Management

Not all product teams initially prioritized integration. I presented data showing 75% of connection-related support tickets came from their apps, demonstrated the prototype's improved experience, and offered implementation support. This secured commitment from all 7 teams within 2 weeks.

Technical Challenge

Backend teams had different connectivity architectures. I facilitated a technical design session with all backend engineers, identified commonalities 80% could leverage, and designed a flexible component API allowing customization while maintaining consistency. The component successfully adapted to 7 different backend architectures.

Design System Contribution

I integrated the component into the centralized Figma library with comprehensive usage documentation, video walkthroughs, and contribution guidelines. This established reusable patterns that reduced implementation time from 2 weeks to 2 days per app.

Impact: The component became the template for other status indicators (loading, errors) and a case study in the design system documentation. Post-launch adoption by 2 additional apps demonstrated its value, and the process became the standard for future component development.

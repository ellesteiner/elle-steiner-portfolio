Design System Leadership

Building consistency and efficiency at scale

Role: Senior UX Designer & Design System Lead  //  Duration: 6 months  //  Team: 2 direct reports + 5 engineers

Impact at a Glance

Reduced Design-to-Dev time by 40%

Increased component reuse from 15% to 75%

Decreased UI inconsistencies by 85% across 7 applications

Scaled system from 7 to 9 apps without additional design resources

The Challenge

Seven Air Force apps operated independently with no shared design standards, resulting in 17 different button variations, inconsistent interactions, and fragmented user experiences. Users struggled to transfer knowledge between apps, while Designers and Engineers duplicated effort solving the same problems repeatedly. I built the case for a comprehensive Design System by demonstrating that inconsistency was costing the organization in rework and inefficiency.

My Approach

Research & Strategic Foundation

I conducted a comprehensive audit across all 7 apps, documenting every component, pattern and interaction. The findings were striking: 17 button variations, 12 different navigation patterns, and few shared components. I synthesized this research into a compelling business case showing how standardization would eliminate duplicate work and accelerate product development.

Key Research Findings:

  • 17 unique button styles across 7 apps (should be 1 flexible component)
  • Design-to-Dev hand-off averaging 2 weeks per feature due to lack of standards
  • 85% of components were recreated multiple times across teams
  • User testing revealed confusion when switching between apps

Design Principles Established:

Based on user research and technical constraints, I established four core principles that would guide all design system decisions:

  • Flexibility with Constraints — Components must adapt to product needs while maintaining consistency
  • Accessibility First — WCAG 2.1 AA compliance as baseline, not afterthought
  • Progressive Enhancement — Basic functionality works everywhere, enhanced features where supported
  • Documentation as Design — If it's not documented, it doesn't exist

Component Architecture & Design Strategy

I designed a three-tier component system that balanced flexibility with consistency:

  • Foundation Layer: Design styles (color, typography, spacing, elevation) providing single source of truth
  • Component Layer: 45 core components with customizable properties via Figma overrides
  • Pattern Layer: Documented interaction patterns and page templates for complex workflows

Override Architecture Strategy: Each component includes carefully designed override options that allow customization without breaking consistency.

For example, the button component includes 8 variants (primary, secondary, tertiary, disabled, size, link, icon-only, split) with configurable sizes, states, and icon placement, but maintains consistent spacing, typography, and interaction behaviors across all variants.

This architecture enabled teams to customize components for their specific needs (e.g., domain-specific icons, contextual labels) while preventing the fragmentation that created 17 different buttons in the first place.

Design System Documentation

I created comprehensive documentation that goes beyond visual specs to include interaction behaviors, accessibility requirements, usage guidelines, and code examples. The documentation answers three questions for every component:

  • When to use this (and when not to)
  • How to implement this (design and code specs)
  • Why it works this way (design decisions and rationale)

Example - Button Documentation Structure:

  • Visual specifications (spacing, typography, colors, states)
  • Interaction specifications (hover, focus, active, disabled, loading)
  • Accessibility requirements (contrast ratios, keyboard navigation, screen reader labels)
  • Usage guidelines (button hierarchy, when to use primary vs. secondary)
  • Code examples (React component with props)
  • Design decisions log (why we chose these variants, what we rejected and why)

The "why" documentation proved invaluable, it prevented teams from making changes that would undermine the system and provided context when revisiting designs months later.

Governance Model Design

I designed a sustainable governance structure that would outlive my direct involvement:

Contribution Process:

  1. Team identifies need for new component or pattern
  2. Submit proposal with use cases and justification
  3. Design System team reviews for duplication/fit
  4. If approved, collaborative design sprint with proposing team
  5. Component added to library with full documentation

Decision Framework:

I established clear criteria for design decisions to reduce ambiguity and empower teams:

  • Consistency vs. Flexibility: Default to consistency; flex only when user research supports it
  • Accessibility: Non-negotiable WCAG 2.1 AA minimum
  • Technical Feasibility: If it can't be built consistently, redesign it
  • Scalability: Will this work when we have 20 apps instead of 7?

This framework enabled faster decisions and reduced the need for escalation.

Adoption Strategy

I designed a phased rollout that reduced resistance and built momentum:

  • Phase 1: Pilot with 2 engaged teams, surface issues, iterate quickly
  • Phase 2: Expand to 3 more teams with documented best practices from pilots
  • Phase 3: Mandate adoption for remaining teams with proven success stories
  • Phase 4: Extend to new applications as they launch

I created training materials including video walk-throughs, live workshops and office hours for implementation support. This reduced integration time from 2 weeks to 2 days per application.

Implementation & Governance

Collaboration with Engineering

I partnered with the 5-person development team to build the component library in code, using MUI as a foundation with extensive customization. I created a parallel structure between Figma and code, ensuring Designers and Developers used the same language and mental models.

Design-Dev Process:

  • Weekly design reviews where developers provided technical feasibility feedback
  • Shared documentation in GitLab with design specs and code examples
  • QA process where I reviewed implemented components against design specs
  • Fast iteration cycle: design → review → implement → QA → deploy

When technical constraints conflicted with design vision (e.g., a team's backend couldn't support real-time connection status), I redesigned solutions that worked within constraints rather than forcing impossible implementations. This pragmatic approach built trust and resulted in higher-quality outcomes.

Outcomes & Impact

Measurable Results

Efficiency Gains:

  • 40% reduction in Design-to-Development hand-off time
  • Component reuse increased from 15% to 75%
  • Design iteration cycles reduced from weeks to days

Quality Improvements:

  • 85% decrease in UI inconsistencies across applications
  • User satisfaction scores improved 25% due to consistent experiences
  • Accessibility compliance increased from 60% to 98%

Scalability:

  • System supported growth from 7 to 9 applications without additional design resources
  • New applications integrated in 2 days vs. 2 weeks previously

System Evolution

The Design System continues to evolve based on product needs:

  • 45 components at launch → 49 components after 12 months
  • 2 teams contributing new components monthly
  • Component library downloads increased 300% as adoption grew
  • System became model for other cross-functional initiatives

Design System Principles in Action

What Makes This System Work

  1. Flexible Architecture — Components adapt to product needs while maintaining consistency. The override system provides 80% of customization needs without breaking the design language.
  2. Documentation as Design — Every component includes the "why" behind design decisions. This prevents regression and builds shared understanding across teams.
  3. Progressive Enhancement — Basic functionality works everywhere. Teams can enhance based on their technical capabilities and user needs.
  4. Accessibility Baked In — WCAG compliance isn't optional, it's built into every component from the start. Color, typography and interaction design all meet AA standards.
  5. Living System — The Design System evolves with product needs through clear contribution processes and governance. It's not a static artifact but a collaborative practice.

Key Learnings

  • Design with Constraints: Understanding technical limitations early prevented costly rework. The best designs are feasible designs.
  • Document Decisions: Recording the "why" behind design choices proved as valuable as the designs themselves. Context prevents future mistakes.
  • System Thinking: Every component affects the whole system. Small inconsistencies compound into large problems at scale.
  • Collaboration Creates Buy-in: Involving teams in design decisions created ownership and smoother adoption than mandating top-down standards.

Conclusion

This Design System demonstrates my approach to complex UX challenges:

  • Systematic research
  • Strategic design thinking
  • Collaborative execution
  • Measurable impact

By establishing shared standards across 7 applications, I eliminated duplicate work, improved user experience consistency, and enabled the organization to scale efficiently.

The system's continued evolution and adoption beyond its original scope validates the foundational architecture and governance model, proof that thoughtful systems design creates lasting value.

Creating a Figma Component

Figma is an excellent platform for creating and managing the components that make up a Design System. While Figma itself isn't a Design System, it serves as a powerful tool that enables teams to build, organize, and maintain one effectively. Figma allows for:

  • Streamlined Collaboration: Figma's real-time collaboration allows multiple designers, developers and stakeholders to work simultaneously on the Design System, fostering efficient communication and feedback loops.
  • Component Libraries and Style Management: Create, organize and reuse design components (buttons, forms, etc.) and styles (colors, fonts, spacing) across projects, ensuring consistency and reducing design time.
  • Version Control and History: Track changes, iterate on components and revert to previous versions easily, maintaining transparency and facilitating collaboration.
  • Easy Handoff and Development: Figma generates code snippets and asset exports for developers, enabling smooth hand-off and accelerating development.

Figma allows designers to create components, such as buttons, that can be customized through selectable overrides. During this phase, I collaborated with two other designers to completely rebuild all components within a new Figma component library. We included a full range of style options, enabling teams to easily customize components to fit the specific needs of their applications.

Collaborating with Development to Bring the Design System to Life

Once all the components were rebuilt in Figma, it was time to bring the development teams into the process. Our Design System combined customized MUI components with fully custom-built components tailored to our organization's needs. We partnered closely with the developers to ensure that all MUI-based components were styled and functioned according to spec. We also collaborated with them to build and refine the custom components to maintain consistency with the design standards.

By the end of the process, we had a fully developed library of coded components that engineers could easily integrate into their projects.
